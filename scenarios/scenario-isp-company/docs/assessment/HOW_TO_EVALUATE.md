# Как проверить результаты обучения по сценарию

## Обзор процесса оценки

Данное руководство предназначено для преподавателей и описывает различные методы проверки результатов обучения студентов по сценарию "ISP компания".

---

## Методы проверки результатов

### 1. Проверка чек-листов (CHECKLISTS.md)

**Назначение**: Структурированная проверка выполнения задач

**Процесс**:
1. Попросите студентов заполнить чек-листы самостоятельно
2. Проверьте заполненные чек-листы
3. Сравните с правильными ответами из SOLUTIONS.md
4. Выставьте баллы согласно критериям

**Преимущества**:
- Быстрая проверка
- Объективная оценка
- Легко отслеживать прогресс

**Недостатки**:
- Студенты могут отмечать невыполненные задачи
- Требует дополнительной верификации

**Рекомендации**:
- Используйте чек-листы как основу, но проверяйте фактическое выполнение
- Попросите студентов предоставить доказательства (скриншоты, логи)

---

### 2. Проверка отчетов об инцидентах (REPORT_TEMPLATE.md)

**Назначение**: Оценка навыков документирования и анализа

**Процесс**:
1. Попросите студентов заполнить шаблон отчета
2. Проверьте качество описания уязвимостей
3. Оцените правильность методологии
4. Проверьте наличие доказательств (скриншоты, логи)
5. Оцените предложенные исправления

**Критерии оценки**:

| Критерий | Вес | Описание |
|----------|-----|----------|
| Полнота описания | 25% | Все уязвимости описаны подробно |
| Качество доказательств | 25% | Скриншоты, логи, команды |
| Правильность методологии | 20% | Корректное использование инструментов |
| Качество исправлений | 20% | Правильные и полные исправления |
| Структура и оформление | 10% | Следование шаблону, грамотность |

**Преимущества**:
- Оценивает навыки документирования
- Показывает понимание процесса
- Можно использовать для портфолио

**Недостатки**:
- Требует больше времени на проверку
- Субъективная оценка качества

---

### 3. Практическая демонстрация

**Назначение**: Проверка реальных навыков выполнения задач

**Процесс**:
1. Попросите студента продемонстрировать выполнение задачи
2. Наблюдайте за процессом в реальном времени
3. Задавайте вопросы о выбранных методах
4. Оцените правильность выполнения

**Формат**:
- **Очная демонстрация**: Студент показывает на своем компьютере
- **Удаленная демонстрация**: Через screen sharing (Zoom, Teams)
- **Запись экрана**: Студент записывает видео выполнения

**Критерии оценки**:

| Критерий | Описание |
|----------|----------|
| Правильность команд | Использование правильных команд и инструментов |
| Понимание процесса | Объяснение своих действий |
| Эффективность | Минимум ошибок, правильная последовательность |
| Решение проблем | Способность решать возникающие проблемы |

**Преимущества**:
- Видно реальные навыки
- Можно задать вопросы
- Сложно списать

**Недостатки**:
- Требует много времени
- Нужна техническая инфраструктура

### 4. Автоматизированная проверка

**Назначение**: Быстрая проверка исправлений уязвимостей

**Методы**:

#### 4.1 Проверка через Ansible

Создайте playbook для автоматической проверки исправлений:

```yaml
# verify-fixes.yml
- name: Проверка исправлений уязвимостей
  hosts: all
  tasks:
    - name: Проверка пароля MySQL
      mysql_user:
        login_host: "{{ ansible_host }}"
        login_user: root
        login_password: "{{ mysql_root_password }}"
        name: radius
        password: "{{ mysql_radius_password }}"
      register: mysql_check
      
    - name: Проверка конфигурации FreeRADIUS
      command: grep -q "rate_limit" /etc/freeradius/3.0/sites-available/default
      register: radius_config
      
    - name: Проверка обновлений системы
      command: apt list --upgradable
      register: updates
```

#### 4.2 Проверка через скрипты

**Готовый скрипт**: `infrastructure/scripts/verify-fixes.sh`

Скрипт автоматически проверяет исправления на всех машинах сценария:

```bash
# Проверка всех машин
./infrastructure/scripts/verify-fixes.sh

# Проверка только RADIUS сервера
./infrastructure/scripts/verify-fixes.sh -m radius-server

# Подробный вывод
./infrastructure/scripts/verify-fixes.sh -v

# Сохранение результатов в файл
./infrastructure/scripts/verify-fixes.sh -o results.txt
```

**Что проверяет скрипт**:
- RADIUS сервер: пароль MySQL, rate limiting, шифрование паролей
- Биллинговый сервер: SQL инъекции, пароль PostgreSQL, prepared statements
- Веб-сервер: версия WordPress, защита от XSS, конфигурация Apache
- Сервер мониторинга: SNMP community strings, доступ к Zabbix
- Jump-сервер: SSH аутентификация, защита от брутфорса
- Компьютер администратора: SMB папки, антивирус

**Пример вывода**:
```
✓ SSH подключение
✓ Пароль MySQL изменен
✗ Rate limiting в FreeRADIUS
  → Rate limiting не настроен
✓ Системные обновления установлены

Итоговая статистика:
Всего проверок: 5
Успешно: 3
Провалено: 2
Процент успешных проверок: 60%
Оценка: Удовлетворительно (3)
```

**Создание собственных скриптов**:

```bash
#!/bin/bash
# verify-radius-fixes.sh

# Проверка пароля MySQL
if mysql -h 192.168.10.10 -u radius -p"radius123" -e "SELECT 1" 2>/dev/null; then
    echo "❌ Слабый пароль MySQL не изменен"
    exit 1
else
    echo "✅ Пароль MySQL изменен"
fi

# Проверка rate limiting в FreeRADIUS
if grep -q "rate_limit" /etc/freeradius/3.0/sites-available/default; then
    echo "✅ Rate limiting включен"
else
    echo "❌ Rate limiting не включен"
    exit 1
fi
```

#### 4.3 Проверка через сканирование

Используйте автоматизированные сканеры для проверки:

```bash
# Проверка, что уязвимости устранены
nmap --script vuln 192.168.10.10
nikto -h http://192.168.20.10
sqlmap -u "http://192.168.20.10/login.php" --batch --crawl=2
```

**Преимущества**:
- Быстрая проверка
- Объективная оценка
- Можно проверить всех студентов одновременно

**Недостатки**:
- Требует настройки автоматизации
- Не проверяет понимание процесса

### 5. Устный опрос (Viva)

**Назначение**: Проверка понимания концепций и методов

**Процесс**:
1. Задайте вопросы о найденных уязвимостях
2. Попросите объяснить выбранные методы
3. Обсудите альтернативные подходы
4. Проверьте понимание последствий

**Примеры вопросов**:
- Почему вы выбрали именно этот инструмент?
- Какие альтернативные методы могли бы использовать?
- Какие последствия могла бы иметь эта уязвимость в реальной системе?
- Как бы вы предотвратили подобные уязвимости в будущем?

**Критерии оценки**:

| Критерий | Описание |
|----------|----------|
| Понимание концепций | Правильное объяснение методов и техник |
| Обоснованность выбора | Логичное объяснение выбора инструментов |
| Понимание рисков | Осознание последствий уязвимостей |
| Критическое мышление | Способность анализировать и предлагать решения |

**Преимущества**:
- Проверяет глубокое понимание
- Можно задать уточняющие вопросы
- Оценивает коммуникативные навыки

**Недостатки**:
- Требует много времени
- Субъективная оценка
- Может быть стрессовым для студентов

## Комплексная система оценки

### Рекомендуемый подход

Используйте комбинацию методов для всесторонней оценки:

1. **Чек-листы (30%)** - базовая проверка выполнения
2. **Отчеты (30%)** - навыки документирования
3. **Практическая демонстрация (25%)** - реальные навыки
4. **Автоматизированная проверка (10%)** - объективная верификация
5. **Устный опрос (5%)** - понимание концепций

### Шкала оценки

| Оценка | Процент | Описание |
|--------|---------|----------|
| Отлично (5) | 90-100% | Все задачи выполнены, глубокое понимание |
| Хорошо (4) | 75-89% | Большинство задач выполнено, хорошее понимание |
| Удовлетворительно (3) | 60-74% | Основные задачи выполнены, базовое понимание |
| Неудовлетворительно (2) | <60% | Задачи не выполнены или выполнены неправильно |

## Инструменты для проверки

### 1. Ansible для автоматизации

Создайте playbook для проверки всех исправлений:

```yaml
# assessment/verify-all-fixes.yml
- name: Комплексная проверка исправлений
  hosts: all
  tasks:
    - include_tasks: verify-radius.yml
    - include_tasks: verify-billing.yml
    - include_tasks: verify-web.yml
    - include_tasks: verify-monitoring.yml
```

### 2. Скрипты проверки

**Готовый скрипт**: `infrastructure/scripts/verify-fixes.sh`

Универсальный скрипт для проверки всех машин:

```bash
# Проверка всех машин
./infrastructure/scripts/verify-fixes.sh

# Проверка конкретной машины
./infrastructure/scripts/verify-fixes.sh -m radius-server

# С сохранением результатов
./infrastructure/scripts/verify-fixes.sh -o results.txt
```

**Документация**: См. `infrastructure/scripts/VERIFY_FIXES_README.md`

**Создание собственных скриптов**:

```bash
assessment/
├── verify-radius.sh      # Проверка RADIUS сервера
├── verify-billing.sh     # Проверка биллинга
├── verify-web.sh         # Проверка веб-сервера
├── verify-monitoring.sh  # Проверка мониторинга
└── verify-all.sh         # Проверка всех систем
```

## Процесс проверки по этапам

### Этап 1: Предварительная проверка

1. **Проверка развертывания**:
   - Все ли машины развернуты?
   - Работают ли сервисы?
   - Доступны ли машины по сети?

2. **Проверка базовых навыков**:
   - Может ли студент подключиться к машинам?
   - Понимает ли сетевую топологию?
   - Знает ли IP адреса машин?

### Этап 2: Проверка обнаружения уязвимостей

1. **Проверка чек-листов разведки**:
   - Обнаружены ли все открытые порты?
   - Определены ли версии сервисов?
   - Найдены ли уязвимости?

2. **Проверка доказательств**:
   - Есть ли скриншоты сканирования?
   - Сохранены ли логи?
   - Документированы ли находки?

### Этап 3: Проверка эксплуатации

1. **Проверка успешной эксплуатации**:
   - Получен ли доступ к системам?
   - Извлечены ли данные?
   - Документированы ли результаты?

2. **Проверка понимания**:
   - Понимает ли студент, что он делает?
   - Может ли объяснить процесс?
   - Знает ли последствия?

### Этап 4: Проверка исправлений

1. **Автоматизированная проверка**:
   - Запустите скрипты проверки
   - Проверьте через Ansible
   - Используйте сканеры

2. **Ручная проверка**:
   - Проверьте конфигурационные файлы
   - Проверьте пароли
   - Проверьте обновления

3. **Валидация**:
   - Попробуйте эксплуатировать уязвимости снова
   - Убедитесь, что они действительно исправлены

## Типичные ошибки студентов

### 1. Неполное выполнение задач
- **Проблема**: Студент отмечает задачу как выполненную, но не доводит до конца
- **Решение**: Требуйте доказательства выполнения

### 2. Копирование решений
- **Проблема**: Студенты копируют решения друг у друга
- **Решение**: Используйте практическую демонстрацию и устный опрос

### 3. Непонимание процесса
- **Проблема**: Студент выполняет команды, но не понимает, что делает
- **Решение**: Задавайте вопросы о выбранных методах

### 4. Неправильные исправления
- **Проблема**: Студент "исправляет" уязвимость, но делает это неправильно
- **Решение**: Используйте автоматизированную проверку и валидацию

## Рекомендации для эффективной проверки

### 1. Подготовка

- Заранее подготовьте все инструменты проверки
- Протестируйте скрипты автоматизации
- Подготовьте вопросы для устного опроса

### 2. Организация

- Установите четкие сроки сдачи работ
- Определите формат предоставления результатов
- Создайте систему для отслеживания прогресса

### 3. Обратная связь

- Предоставляйте детальную обратную связь
- Указывайте на конкретные ошибки
- Предлагайте способы улучшения

### 4. Честность

- Используйте разные варианты заданий
- Меняйте IP адреса и конфигурации
- Проводите практические демонстрации

## Заключение

Эффективная проверка результатов обучения требует комбинации различных методов. Используйте автоматизацию для быстрой проверки, но не забывайте о практических демонстрациях и устных опросах для оценки глубокого понимания.

**Помните**: Цель не только проверить выполнение задач, но и убедиться, что студенты понимают процессы и могут применять знания в реальных ситуациях.

## 1. ВВЕДЕНИЕ

### 1.1. Актуальность темы

В современном мире количество кибератак и уязвимостей неуклонно растёт. Многие инциденты происходят не из-за сложных хакерских атак, а из-за человеческих ошибок — неправильных настроек, устаревшего программного обеспечения, слабых паролей и отсутствия базовых мер защиты.

Анализ инцидентов информационной безопасности показывает, что большинство из них происходят из-за:

- **Эксплуатации давно известных уязвимостей** — многие организации не успевают применять обновления безопасности своевременно
- **Человеческого фактора** — ошибок администрирования, связанных с недостаточной квалификацией или невнимательностью
- **Некорректной конфигурации систем** — неправильных настроек безопасности, открытых портов, слабых паролей

Особенно уязвимы небольшие организации, где часто нет выделенных специалистов по безопасности, и администрирование выполняется «по остаточному принципу». Такие ошибки приводят к серьёзным последствиям — утечкам данных и потере работоспособности систем.

Для качественной подготовки специалистов по кибербезопасности недостаточно только теории — необходимо отрабатывать практические сценарии, максимально приближенные к реальным условиям. Подготовка специалистов по кибербезопасности требует практических навыков защиты операционных систем. 

Киберполигоны позволяют безопасно моделировать такие ситуации, не затрагивая реальные сети. Создание изолированной учебной площадки (киберполигона) позволяет безопасно отрабатывать реальные сценарии атак и защиты, не затрагивая рабочие сети. Именно поэтому создание учебных инфраструктур, имитирующих типичные ошибки и уязвимости, является актуальной задачей.

### 1.2. Цель практики

Цель данной работы заключается в разработке реалистичных учебных инфраструктур для курса «Защита ОС», имитирующих работу небольших организаций с типовыми нарушениями и ошибками администрирования, с возможностью автоматизированного развертывания и использования на Киберполигоне ТюмГУ.

Для достижения поставленной цели разработанная платформа должна обеспечивать:

- **Автоматизированное развертывание учебных сценариев** — использование принципов Infrastructure as Code (Terraform, Ansible) для быстрого и единообразного создания лабораторных стендов
- **Моделирование типичных ошибок администрирования и уязвимостей** — реалистичное воспроизведение проблем безопасности, характерных для небольших организаций
- **Безопасную изолированную среду для обучения** — полная изоляция учебных сред от производственных систем
- **Интеграцию с существующей инфраструктурой** — возможность использования на Киберполигоне ТюмГУ
- **Инструменты для оценки результатов обучения** — автоматизированная проверка исправлений и методические материалы

### 1.3. Задачи практики

Для достижения поставленной цели в ходе практики необходимо решить следующие задачи:

1. **Определить типовые уязвимости и ошибки администрирования, характерные для небольших организаций** — выявить наиболее распространенные проблемы безопасности, связанные с человеческим фактором, некорректной конфигурацией систем и эксплуатацией известных уязвимостей

2. **Разработать учебные инфраструктуры, моделирующие конфигурационные ошибки и угрозы безопасности в небольших организациях** — создать реалистичные сценарии, имитирующие работу небольших организаций с типичными нарушениями и ошибками администрирования

3. **Реализовать автоматизированное развертывание инфраструктур с использованием принципов IaC** — создать модули Terraform и Ansible playbooks для автоматизации развертывания учебных сценариев на Киберполигоне ТюмГУ

4. **Подготовить документацию, каталог уязвимостей и методические материалы для обучения** — создать техническую документацию, каталог знаний по уязвимостям, чек-листы для проверки, шаблоны отчетов и решения задач для преподавателей и студентов

5. **Оценить эффективность обучения по разработанным сценариям** — провести тестирование работоспособности платформы и подтвердить эффективность использования сценариев в учебном процессе курса «Защита ОС»

### 1.4. Объект и предмет исследования

**Объект исследования:** Процесс обучения защите операционных систем в изолированной среде киберполигона.

**Предмет исследования:** Автоматизированная инфраструктура для развертывания учебных сценариев киберучений с типичными ошибками администрирования на Киберполигоне ТюмГУ.

### 1.5. Практическая значимость

Разработанная платформа позволит:

- **Повысить качество практической подготовки специалистов по защите ОС** — студенты получат возможность отрабатывать навыки на реалистичных сценариях
- **Унифицировать учебный процесс** — единообразный подход к созданию и использованию лабораторных стендов
- **Сократить время на создание и обновление лабораторных стендов** — автоматизация развертывания значительно ускоряет подготовку учебных сред
- **Безопасно отрабатывать реальные сценарии атак и реагирования** — изолированная среда исключает риск для производственных систем

## 2. ОБЗОР ПРЕДМЕТНОЙ ОБЛАСТИ

### 2.1. Понятие Cyber Range

Cyber Range (киберполигон) — это изолированная среда, предназначенная для безопасного обучения, тренировки и оценки навыков кибербезопасности. Киберполигоны позволяют моделировать реальные сетевые инфраструктуры с различными уязвимостями и угрозами, не подвергая риску реальные системы.

Основные характеристики Cyber Range:

- **Изоляция:** Полная изоляция от производственных систем
- **Реалистичность:** Моделирование реальных инфраструктур и угроз
- **Масштабируемость:** Возможность создания различных сценариев
- **Автоматизация:** Автоматическое развертывание и управление средами
- **Оценка:** Инструменты для оценки результатов обучения

### 2.2. Технологии виртуализации

Для создания изолированных сред используются различные технологии виртуализации:

**1. Proxmox VE**
- Платформа виртуализации на базе KVM и LXC
- Веб-интерфейс для управления
- Поддержка API для автоматизации
- Бесплатное решение с открытым исходным кодом

**2. Контейнеризация (Docker, LXC)**
- Легковесная виртуализация
- Быстрое развертывание
- Эффективное использование ресурсов

**3. Облачные платформы**
- AWS, Azure, Google Cloud
- Высокая масштабируемость
- Платные решения

### 2.3. Инфраструктура как код (IaC)

Infrastructure as Code (IaC) — это подход к управлению инфраструктурой, при котором конфигурация описывается в виде кода. Основные инструменты:

**Terraform:**
- Декларативное описание инфраструктуры
- Поддержка множества провайдеров
- Версионирование конфигураций
- Планирование изменений

**Ansible:**
- Конфигурационное управление
- Идемпотентность операций
- Простота использования
- Поддержка различных ОС

### 4.4. Сценарий обучения

В качестве основного реализован сценарий **"Защита операционных систем"** (`scenario-os-training`) — учебная инфраструктура для практического обучения навыкам защиты операционных систем. Сценарий включает три виртуальные машины с различными операционными системами, настроенными с типичными уязвимостями ОС.

**Учебная сеть (192.168.100.0/24):**
Все три машины находятся в одной изолированной сети для демонстрации межсистемных уязвимостей и сетевых атак:

- **Ubuntu Server** (192.168.100.10) — демонстрация уязвимостей Linux ОС
  - 12 категорий ошибок безопасности
  - Уязвимости аутентификации, SSH, файловой системы, обновлений и др.
  
- **Windows Server 2016** (192.168.100.20) — демонстрация уязвимостей Windows Server
  - 10 категорий ошибок безопасности
  - Уязвимости GPO, AD, сетевых протоколов, контроллеров домена и др.
  
- **Windows 10 Pro** (192.168.100.30) — демонстрация уязвимостей рабочих станций Windows
  - Клиентские уязвимости ОС
  - Отключенные механизмы защиты, слабые пароли, небезопасные настройки

## 5. РЕАЛИЗАЦИЯ ПЛАТФОРМЫ

### 5.1. Terraform конфигурация для Proxmox

Для обеспечения автоматизированного развертывания на Киберполигоне ТюмГУ разработана Terraform конфигурация для создания виртуальных машин в Proxmox VE:

**Реализованные компоненты:**
- **Ubuntu Server** — автоматическое развертывание через Terraform
  - Использование провайдера `telmate/proxmox` версии 3.0.2-rc06
  - Клонирование из шаблона Ubuntu 20.04
  - Настройка cloud-init для начальной конфигурации
  - Автоматическая настройка сети (IP: 192.168.100.10/24)
  - Интеграция SSH ключей для безопасного доступа

**Основные параметры конфигурации:**
- Имя виртуальной машины: `ubuntu-server-training`
- Ресурсы: 2 CPU cores, 4 GB RAM, 50 GB диск
- Сетевые настройки: IP 192.168.100.10/24, шлюз 192.168.100.1, DNS 8.8.8.8
- SSH ключи для доступа через cloud-init
- Использование virtio для сетевого интерфейса

**Особенности реализации:**
- Использование cloud-init для начальной настройки системы
- Автоматическая настройка статического IP адреса
- Поддержка SSH ключей для безопасного доступа
- Интеграция с существующей инфраструктурой Proxmox VE

**Примечание:** Windows машины (Windows Server 2016 и Windows 10 Pro) создаются вручную через веб-интерфейс Proxmox, так как требуют ручной установки ОС. Terraform конфигурация предоставляет выходные значения с IP адресами для интеграции с Ansible.

### 5.2. Ansible конфигурация

Создан комплексный Ansible playbook (`playbook.yml`) для автоматизированной настройки уязвимостей ОС на всех машинах инфраструктуры. Playbook содержит более 1200 строк кода и реализует полный спектр уязвимостей.

**Для Ubuntu Server (12 категорий уязвимостей):**

**1. Ошибки в управлении аккаунтами и аутентификацией:**
- Создание пользователей со слабыми паролями (admin/admin123, user1/password123, user2/123456)
- Отключение PAM-ограничений для паролей
- Отключение политики блокировки после неверных попыток
- Настройка паролей без срока действия
- Широкие sudo права (NOPASSWD для всех)
- Отсутствие MFA

**2. Уязвимости при конфигурации SSH:**
- Разрешен парольный вход без ограничений
- Разрешен root login
- Слабые алгоритмы шифрования
- Отключен fail2ban и firewall
- Небезопасные настройки authorized_keys

**3. Ошибки конфигурации сети и firewall:**
- Полностью отключен UFW
- Открытые небезопасные порты (Telnet, FTP, Rlogin)
- Отсутствие сетевой сегментации

**4. Ошибки при настройке обновлений и пакетов:**
- Отключены автоматические обновления безопасности
- Небезопасные PPA и репозитории
- Отключена проверка подписей пакетов

**5. Ошибки в управлении службами и демонами:**
- Включены лишние службы (CUPS, Avahi, Bluetooth)
- Службы работают от root
- Отключен AppArmor

**6. Ошибки с правами доступа и файлами:**
- Неправильные права на системные директории (777)
- Широкие sudo права
- Небезопасные cron задачи
- **10 подкатегорий уязвимостей /etc/shadow:**
  - Неправильные права доступа
  - Утечки через бэкапы
  - Ошибки в конфигурации PAM
  - Уязвимые алгоритмы хеширования
  - Неправильные настройки shadow aging
  - Доступ через SUID-приложения
  - Недостаточная защита от локального пользователя
  - Утечка через web-приложения
  - Смешивание system и application accounts
  - Ошибки при клонировании систем

**7. Уязвимости веб-сервисов и приложений:**
- Directory listing включен
- Раскрытие версии сервера

### 6.2. Тестирование уязвимостей

**Цель:** Убедиться, что уязвимости ОС корректно внедрены и могут быть обнаружены.

**Методы тестирования:**

**Для Ubuntu Server:**
1. Проверка слабых паролей (ssh admin@192.168.100.10 с паролем admin123)
2. Проверка разрешенного root login (ssh root@192.168.100.10)
3. Проверка прав доступа к файлам (ls -la /etc, /var, /home)
4. Проверка статуса UFW (ufw status)
5. Проверка статуса fail2ban (systemctl status fail2ban)
6. Проверка конфигурации SSH (grep PermitRootLogin /etc/ssh/sshd_config)
7. Проверка прав на /etc/shadow (ls -la /etc/shadow)
8. Поиск учетных данных в открытом виде (find / -name "*credential*")
9. Проверка включенных уязвимых протоколов (SMBv1)
10. Проверка статуса обновлений (cat /etc/apt/apt.conf.d/20auto-upgrades)

**Для Windows Server 2016 и Windows 10 Pro:**
1. Проверка слабых паролей (RDP подключение с Admin123!)
2. Проверка политики паролей (Get-ItemProperty HKLM:\SYSTEM\CurrentControlSet\Services\Netlogon\Parameters)
3. Проверка статуса брандмауэра (Get-NetFirewallProfile)
4. Проверка статуса Windows Defender (Get-MpPreference)
5. Проверка SMBv1 (Get-WindowsOptionalFeature -Online -FeatureName SMB1Protocol)
6. Проверка NTLM (Get-ItemProperty HKLM:\SYSTEM\CurrentControlSet\Control\Lsa)
7. Проверка PowerShell Remoting (Get-Item WSMan:\localhost\Client\TrustedHosts)
8. Проверка статуса UAC (Get-ItemProperty HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Policies\System)

**Результаты:**
- ✅ Все 12 категорий уязвимостей Ubuntu успешно внедрены
- ✅ Все 10 категорий уязвимостей Windows успешно внедрены
- ✅ Уязвимости могут быть обнаружены стандартными инструментами

- ✅ Все машины находятся в одной сети и могут взаимодействовать

### Приложение В. Пример Ansible playbook

```yaml
---
# Ansible playbook для настройки уязвимостей ОС в учебной инфраструктуре
# Фокус только на уязвимостях операционных систем

- name: Настройка Ubuntu Server с уязвимостями ОС Linux
  hosts: ubuntu-server
  become: yes
  gather_facts: yes
  
  tasks:
    # 1. ОШИБКИ В УПРАВЛЕНИИ АККАУНТАМИ И АУТЕНТИФИКАЦИЕЙ
    - name: Создание пользователей со слабыми паролями
      user:
        name: "{{ item.name }}"
        password: "{{ item.password | password_hash('sha512') }}"
        shell: /bin/bash
        groups: sudo
        append: yes
        password_expire_max: -1
      loop:
        - { name: "admin", password: "admin123" }
        - { name: "user1", password: "password123" }
        - { name: "user2", password: "123456" }
    
    - name: Отключение PAM-ограничений для паролей
      lineinfile:
        path: /etc/security/pwquality.conf
        regexp: "^{{ item.regexp }}"
        line: "{{ item.line }}"
      loop:
        - { regexp: "^minlen", line: "minlen = 1" }
        - { regexp: "^dcredit", line: "dcredit = 0" }
    
    # 2. УЯЗВИМОСТИ ПРИ КОНФИГУРАЦИИ SSH
    - name: Настройка SSH с небезопасными параметрами
      lineinfile:
        path: /etc/ssh/sshd_config
        regexp: "^{{ item.regexp }}"
        line: "{{ item.line }}"
      loop:
        - { regexp: "^#?PermitRootLogin", line: "PermitRootLogin yes" }
        - { regexp: "^#?PasswordAuthentication", line: "PasswordAuthentication yes" }
        - { regexp: "^#?MaxAuthTries", line: "MaxAuthTries 1000" }
      notify: restart ssh
    
    # 3. ОШИБКИ КОНФИГУРАЦИИ СЕТИ И FIREWALL
    - name: Отключение UFW
      ufw:
        state: disabled
    
    # 4. ОШИБКИ ПРИ НАСТРОЙКЕ ОБНОВЛЕНИЙ
    - name: Отключение автоматических обновлений
      file:
        path: /etc/apt/apt.conf.d/20auto-upgrades
        state: absent
    
    # 5. ОШИБКИ С ПРАВАМИ ДОСТУПА
    - name: Создание директорий с неправильными правами
      file:
        path: "{{ item.path }}"
        state: directory
        mode: '{{ item.mode }}'
      loop:
        - { path: "/srv/public", mode: '0777' }
        - { path: "/srv/sensitive", mode: '0777' }
        - { path: "/home/admin/credentials", mode: '0777' }
    
    # ... (остальные категории уязвимостей)

- name: Настройка Windows Server 2016 с уязвимостями ОС Windows
  hosts: windows-server-2016
  gather_facts: yes
  
  tasks:
    # 1. УЯЗВИМОСТИ АУТЕНТИФИКАЦИИ
    - name: Настройка слабых паролей
      win_user:
        name: "{{ item.name }}"
        password: "{{ item.password }}"
        password_never_expires: yes
      loop:
        - { name: "admin", password: "admin123" }
        - { name: "user1", password: "password123" }
    
    # 2. ОШИБКИ В НАСТРОЙКЕ GPO
    - name: Разрешение PowerShell Remoting без ограничений
      win_shell: |
        Enable-PSRemoting -Force
        Set-Item WSMan:\localhost\Client\TrustedHosts -Value "*"
    
    # ... (остальные категории уязвимостей)
```
